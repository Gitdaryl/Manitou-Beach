## What We Talked About

### SEO Strategy
Started with a question: does optimising the MB site mean every new business/food truck/event listing lifts MB's rankings? Answer: yes - this is the **topical authority** model. Every new listing adds a hyper-local entity (business name + category + location) that Google's Knowledge Graph can associate with Manitou Beach. More listings = broader keyword coverage without writing a single blog post.

Also confirmed: MB listing Holly's real estate in multiple ways (business directory + winery trail + featured slot) means MB ranks when someone searches "real estate Devils Lake MI" - and Holly's listing surfaces in those results. Co-citation / entity association at work.

### The Product Insight
Biggest strategic moment of the session: the profile page is not just an SEO feature - it's a **$9/mo self-service web presence** that replaces Facebook + GoDaddy for small businesses with no website.

An electrician in Tecumseh pays $9/mo and gets:
- A real URL: `manitoubeachmichigan.com/business/jims-electric`
- Hero photo, description, hours, contact details
- Call button, quote request form
- Google Reviews integration (Sprint 3)
- Monthly AI report card showing their traffic (Sprint 4)

The comparison to Facebook is deliberate: it should eventually be as natural as creating a Facebook profile - fully self-service, no Daryl involved. The unlock gate for that is **SMS verification for self-edit** (Sprint 2).

### The Facebook Moment
Three things need to exist before it feels self-service:
1. SMS verify to unlock edit mode
2. Photo upload direct from phone
3. Monthly "your profile got X eyes" SMS so they know it's working

Once those three are live, a business owner can discover the platform, claim their listing, add a photo, fill in their hours, and be live - all without touching Daryl. That's the retention flywheel.

### How slugs work
Businesses have no `slug` field in Notion. The page fetches all businesses via `/api/businesses?all=true`, then filters by matching `toSlug(business.name) === slug`. This means:
- Slug is generated client-side from the name
- If the name changes in Notion, the URL breaks
- Long-term: add a permanent `slug` field to Notion DB

Food trucks already have a `slug` field in Notion - no issue there.

## Key Architecture Notes for Next Session

- **One file rule**: `src/App.jsx` is intentionally monolithic (~10,500 lines). Do NOT refactor into multiple files.
- **Profile pages are separate files**: `src/pages/BusinessProfilePage.jsx` and `src/pages/FoodTruckProfilePage.jsx` - this is the exception (lazy-loaded routes).
- **Winery data**: `WINERY_VENUES` lives in `src/data/wineries.js` - import from there, not from `WineriesPage.jsx` or anywhere else.
- **Design tokens**: Always use `C.` colour tokens (imported from `src/data/config`), never raw hex.
- **CSS for responsive**: Add breakpoints as classes inside a `<style>` tag in the component, not inline.
- **SVG icons in JSX**: Multiple sibling SVG elements as a prop value must be wrapped in a React fragment `<>...</>` or build breaks.
- **Slugify**: `import { toSlug } from '../utils/slugify'` - used anywhere you need name → URL slug.
